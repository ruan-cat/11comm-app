# 2026-04-04 微信小程序轻量壳与 H5 主业务设计方案

## 1. 文档定位与范围

本文档用于收敛当前项目“微信小程序内嵌 H5”方案的架构设计与首轮落地顺序。

当前结论已经明确：

1. 采用“微信小程序轻量壳 + H5 主业务 + Nitro 后端”的组合方案。
2. 第一阶段目标不是做强原生小程序，而是先把现有业务链路稳定跑通。
3. 第一阶段不预留微信支付、订阅消息等未来能力，避免架构提前复杂化。

本文档只描述设计方案、边界划分、迁移顺序与环境策略，不直接包含实现代码。

## 2. 目标摘要

本次方案的目标如下：

1. 让当前 uni-app 项目从“主业务前端”收敛为“微信小程序入口壳”。
2. 让 H5 站点承接绝大多数业务页面、样式、路由和交互逻辑。
3. 继续复用 Nitro 作为统一后端能力，不重建第二套后端。
4. 优先解决当前微信小程序样式兼容差、业务页面维护成本高的问题。
5. 以最小风险完成首批业务链路打通，再逐步补齐 bridge 能力。

## 3. 核心设计决策

### 3.1. 总体架构选型

最终采用如下模型：

```plain
微信小程序壳
  ├─ 首页 / tabBar / 启动承接
  ├─ web-view 容器页
  └─ 少量原生 bridge 页

H5 主业务站
  ├─ 工作台
  ├─ 投诉 / 报修 / 工单
  ├─ 物业 / 费用 / 活动 / 公告 / 报表
  └─ 业务路由与业务 UI

Nitro 后端
  ├─ 业务 API
  ├─ Mock / 测试数据能力
  └─ 后续统一服务端扩展
```

该模型的本质不是“给现有小程序补一个 H5 页”，而是“把小程序项目降级为微信入口容器”。

### 3.2. 小程序壳的职责边界

小程序壳只保留以下职责：

1. 首页与 tabBar。
2. 启动承接与环境识别。
3. 通用 `web-view` 容器页。
4. 少量必须更稳的原生能力页。
5. H5 地址分发和基础参数注入。

小程序壳不再承担以下职责：

1. 大部分业务列表、详情、表单和流程页。
2. 复杂业务样式适配。
3. 多模块业务路由流转。
4. 大量业务状态管理。

### 3.3. H5 主业务站的职责边界

H5 主业务站承接以下内容：

1. 工作台主界面。
2. 投诉、报修、工单、物业、费用、活动、公告、报表等业务页。
3. 大多数业务路由、列表、详情、编辑、提交流程。
4. 大多数业务样式、组件和交互细节。

H5 主业务站不承担以下职责：

1. 小程序首页壳和 tabBar。
2. 小程序原生能力的直接调用细节。
3. 小程序环境差异下的容器级导航责任。

### 3.4. 第一阶段原生能力范围

结合当前仓库已有业务与能力调用，第一阶段建议保留为原生能力的只有以下几类：

1. 扫码。
2. 定位。
3. 图片选择 / 上传。
4. 头像选择。
5. 电话联系。
6. 分享承接。

第一阶段明确不预留：

1. 原生微信支付。
2. 订阅消息。
3. 更复杂的壳内状态同步机制。

### 3.5. 现有仓库的复用策略

当前仓库已经存在可复用基础：

1. [`src/pages/webview/index.vue`](/d:/code/ruan-cat/01s-11comm-app/src/pages/webview/index.vue) 已具备基础容器页形态。
2. [`src/utils/index.ts`](/d:/code/ruan-cat/01s-11comm-app/src/utils/index.ts) 已存在微信小程序环境分流逻辑。
3. 首页与工作台入口配置已集中在 [`src/pages/index/home-menu-config.ts`](/d:/code/ruan-cat/01s-11comm-app/src/pages/index/home-menu-config.ts) 和 [`src/pages/work-dashboard/menu-config.ts`](/d:/code/ruan-cat/01s-11comm-app/src/pages/work-dashboard/menu-config.ts)。

因此本次设计不建议新建第二套 web-view 基建，而是直接升级已有容器页和导航配置。

## 4. 页面结构设计

### 4.1. 壳层页面分层

建议将小程序端页面收敛成 4 层：

1. 壳首页层：首页、tabBar、少量总入口。
2. 容器页层：通用 `web-view` 容器。
3. bridge 页层：扫码、定位、上传、分享承接等少量能力页。
4. H5 主业务层：绝大多数业务页面在 H5 内部完成。

### 4.2. 推荐页面结构

推荐收敛到如下页面结构：

|      页面职责      |          推荐路径           |                        说明                        |
| :----------------: | :-------------------------: | :------------------------------------------------: |
|     小程序首页     |    `/pages/index/index`     |              保留首页与 tabBar 总入口              |
|   通用 H5 容器页   |   `/pages/webview/index`    |         复用现有页面，升级为正式业务容器页         |
|   扫码 bridge 页   |   `/pages/mp-bridge/scan`   |            原生扫码后回传轻量结果给 H5             |
|   定位 bridge 页   | `/pages/mp-bridge/location` |          原生定位后回传经纬度或状态给 H5           |
| 上传 bridge 页可选 |  `/pages/mp-bridge/upload`  | 仅当 H5 在 `web-view` 内上传兼容性不足时再补充实现 |
|   分享承接页可选   |  `/pages/mp-bridge/share`   |     第一阶段可并入容器页或首页，不强制单独拆页     |

### 4.3. 导航原则

后续导航只保留两种主路径：

1. 小程序内部导航：只用于首页、bridge 页、容器页之间切换。
2. H5 内部导航：大部分业务页之间由 H5 路由自身完成。

不建议保留以下混合跳转形态：

```plain
小程序业务页 A -> 小程序业务页 B -> H5 页 C -> 小程序业务页 D
```

推荐收敛为：

```plain
小程序首页 -> web-view -> H5 内部完成业务流
```

## 5. H5 与小程序通信设计

### 5.1. 总体原则

通信设计采用以下原则：

1. 业务状态以后端和 URL 为准。
2. 壳与 H5 之间只传少量控制信息。
3. 不建设复杂实时双向消息总线。
4. 大对象与业务详情一律回到后端重新拉取。

### 5.2. URL 参数职责

URL 参数只承担初始化职责，推荐用于传递：

1. H5 当前业务路由。
2. 页面标题。
3. 来源标记，如 `from=mp-weixin`。
4. 环境标记，如 `env=develop|trial|release`。
5. bridge 启用标记，如 `bridge=1`。

### 5.3. H5 调小程序的调用方式

H5 主动请求小程序能力时，统一通过 `wx.miniProgram` 相关接口触发：

1. `navigateTo`
2. `navigateBack`
3. `redirectTo`
4. `postMessage`
5. `getEnv`

推荐在 H5 内部封装统一桥接对象，例如：

```plain
miniProgramBridge
  ├─ isInMiniProgram()
  ├─ openMiniPage(path, query)
  ├─ navigateBack()
  ├─ openScanPage()
  ├─ openLocationPage()
  └─ postMessage(type, payload)
```

### 5.4. 小程序调 H5 的方式

小程序壳不直接操纵 H5 DOM，也不维护 H5 内部页面状态。小程序对 H5 的影响只保留两种：

1. 改变 `web-view` 的 `src`。
2. 通过 URL 或轻量消息让 H5 自行重新请求数据。

### 5.5. Bridge 调用模型

第一阶段能力建议按以下方式实现：

1. 扫码：H5 发起 -> 小程序扫码页执行 -> 结果回传 H5。
2. 定位：H5 发起 -> 小程序定位页执行 -> 结果回传 H5。
3. 上传：优先尝试 H5 内直传，不稳定时再转原生 bridge 页。
4. 分享：先由小程序壳承接入口，不追求第一阶段全动态分享模型。

### 5.6. 通信层风险控制

需要明确以下限制：

1. `postMessage` 不应当被当作实时双向状态同步通道。
2. 页面标题不要完全依赖运行中动态同步，优先在入口 URL 中携带。
3. 壳与 H5 之间不传递复杂业务对象。
4. bridge 能力优先从轻量控制流实现，不做一次性万能页。

## 6. 业务迁移边界

### 6.1. 优先迁入 H5 的业务模块

基于当前项目页面结构和微信样式兼容现状，建议优先迁入 H5 的业务模块如下：

1. 工作台主业务页。
2. 投诉相关页面。
3. 报修相关页面。
4. 工单相关页面。
5. 物业与费用相关页面。
6. 活动、公告、报表等展示与业务混合页面。

### 6.2. 保留在壳层的业务入口

第一阶段壳层只保留：

1. 首页入口。
2. 工作台入口分发。
3. bridge 能力入口。
4. 极少量容器级兜底页。

### 6.3. 不建议在第一阶段继续扩大的范围

以下方向不建议在第一阶段同步推进：

1. 大量保留原生业务页并与 H5 长期并存。
2. 先做复杂的分享动态化。
3. 先做支付、订阅消息的架构预留。
4. 先清理删除所有旧页面。

## 7. 环境与部署设计

### 7.1. 开发环境策略

开发环境采用“两套链路并行”的现实策略：

1. H5 / Nitro 继续走本地开发链路。
2. 微信小程序壳继续走本地 `mp-weixin` 开发链路。
3. 真机联调优先依赖公网 `https` 测试域名，而不是长期依赖 `localhost` 直连。

该策略的核心目标不是“所有东西都完全本地”，而是“保证微信真机联调稳定”。

### 7.2. 测试环境策略

测试环境建议固定为：

1. H5 测试站。
2. Nitro 测试 API。
3. 微信小程序 `develop / trial` 环境。

后续配置层建议具备以下概念：

1. `H5_BASE_URL__WEIXIN_DEVELOP`
2. `H5_BASE_URL__WEIXIN_TRIAL`
3. `H5_BASE_URL__WEIXIN_RELEASE`

### 7.3. 生产环境策略

生产环境建议保持双域名模型：

1. H5 生产站，例如 `https://m.xxx.com`
2. Nitro API 生产站，例如 `https://api.xxx.com`

小程序壳在生产环境下只负责把业务入口稳定打开到 H5 主站对应页面。

### 7.4. 微信侧硬约束

基于微信官方文档，后续实施必须满足：

1. `web-view` 使用的业务域名必须是公网 `https` 域名。
2. 域名不能是 IP，且需要备案。
3. 业务域名需在小程序后台配置。
4. 若 H5 中嵌入 `iframe`，其域名也需要在白名单内。
5. 个人类型小程序不支持 `web-view` 能力。

## 8. 分阶段落地计划

### 8.1. 第 1 阶段：先把壳跑通

目标：从小程序首页稳定进入 H5 工作台首页。

本阶段建议执行：

1. 升级现有 `web-view` 容器页。
2. 准备 H5 公网测试域名。
3. 配置小程序业务域名。
4. 首页增加单一稳定入口，先只跳 H5 工作台首页。

验收标准：

```plain
小程序首页 -> web-view -> H5 首页
```

链路稳定可用，不白屏、不报域名错误、不出现基础入口失效。

### 8.2. 第 2 阶段：先迁最值钱的业务页

按收益优先而不是按页面数量迁移，推荐顺序如下：

1. 工作台。
2. 投诉。
3. 报修。
4. 工单。
5. 物业与费用。
6. 活动、公告、报表。

### 8.3. 第 3 阶段：补桥接能力

主业务链稳定后，再依次补齐：

1. 扫码 bridge。
2. 定位 bridge。
3. 上传 bridge。
4. 分享承接。

### 8.4. 第 4 阶段：入口收口与删旧

业务迁移稳定后，再逐步：

1. 将首页与工作台入口全部切到 H5。
2. 保留旧页面作为短期兜底。
3. 再逐步清理明显废弃页面和旧跳转逻辑。

## 10. 默认假设与风险

### 10.1. 默认假设

本文档默认以下前提成立：

1. 当前项目具备可部署的 H5 站点。
2. 当前项目具备可部署的 Nitro 后端。
3. 小程序主体类型支持 `web-view`。
4. 第一阶段的目标是跑通现有业务，而不是构建完整原生小程序能力体系。

### 10.2. 已知风险

当前方案存在以下已知风险：

1. `web-view` 体验无法完全替代原生小程序页。
2. 桥接能力如果过早扩张，复杂度会明显上升。
3. 若继续长期并存两套完整业务页，维护成本会再次失控。
4. 若过早删除旧页面，回退成本会升高。

### 10.3. 风险应对原则

对应风险的处理原则如下：

1. 先稳定壳与 H5 主链，不抢做所有原生能力。
2. 先切入口，不先删旧页。
3. 先跑最小闭环，不先追求全覆盖。
4. 先用测试域名稳定联调，不先把时间耗在本机真机直连上。
