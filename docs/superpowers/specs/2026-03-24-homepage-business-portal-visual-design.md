# 2026-03-24 首页业务门户化视觉设计稿

## 1. 背景与目标

当前首页 `http://localhost:3000/#/pages/index/index` 的视觉语言明显落后于工作台页面 `http://localhost:3000/#/pages/work-dashboard/index`。

当前首页的主要问题如下：

- 顶部区域仍是旧式纯色块，缺少工作台已有的渐变氛围和层级。
- 入口区仍以旧式宫格和 PNG 图标为主，卡片感、留白、图标语言和阴影质感都偏弱。
- 模块之间虽然已经有业务分组，但整体更像“堆入口”，不像现代移动业务门户。
- 如果直接照搬工作台的摘要型设计，会依赖待办统计接口；当前项目并不确认存在稳定可用的摘要数据来源。

本次设计的目标不是把首页复制成工作台，而是将首页升级为与工作台同源、但职责更清晰的“业务门户首页”：

- 首页负责快速进入高频业务。
- 工作台负责承载更完整的分类功能索引。
- 两者共享同一套美术语言，但信息节奏不同。

## 2. 设计定稿

本次定稿方案为：

- 首页整体节奏采用 `B2 入口优先`
- 顶部表达方式采用 `H3 轻公告头`

最终结论如下：

- 首页首屏不再优先展示摘要统计。
- 首页首屏优先展示投诉、报修两个高频主入口。
- 顶部区域只承担业务语境提示与品牌氛围，不承担待办统计职责。
- 页面整体风格向工作台看齐，但首页的职责仍然保持“业务门户”而非“个人工作首页”。

## 3. 视觉语言规范

### 3.1. 顶部区域

顶部区域沿用工作台的蓝色渐变头部语言，但表达方式更轻。

设计要求如下：

- 使用蓝色渐变背景，建立与工作台一致的视觉血缘。
- 不使用头像欢迎语，不以“某某，上午好”作为主表达。
- 顶部主标题采用业务导向文案，例如“业务入口首页”“物业业务门户”。
- 顶部副标题用于说明当前首页职责，例如“快速发起投诉、报修等高频业务”。
- 在标题下方放置一条轻量业务提醒带，用于承载“今日重点”“业务提示”“运营文案”。

这条提醒带的定位如下：

- 它是业务语境提示，不是摘要统计卡。
- 它优先支持静态文案、本地常量、运营配置。
- 后续如果存在公告接口，可平滑替换为接口数据。

### 3.2. 主入口区

首页第一视觉重心为两张并列的大卡入口：

- 投诉待办
- 报修待办

设计要求如下：

- 两张卡必须首屏可见。
- 两张卡在尺寸、视觉权重、阴影和圆角上高于其他入口。
- 卡片标题短而明确，直接表达业务动作或业务入口。
- 卡片副文案只说明价值，不承载复杂状态信息。

这两张卡的目标不是展示更多信息，而是降低进入高频业务的点击成本。

### 3.3. 辅助入口区

巡检打卡、设备保养等入口放在第二层级。

设计要求如下：

- 仍然保持可见，避免被折叠过深。
- 视觉权重低于投诉、报修主入口。
- 卡片尺寸可小于主入口卡。
- 与主入口保持同源样式，但弱化阴影、字号或标签强调。

### 3.4. 业务面板区

首页下方继续保留业务分组，但展示方式从旧式硬边框宫格升级为圆角面板式承接。

建议保留的三类业务面板如下：

- 工作待办
- 工作单
- 维修报修

设计要求如下：

- 每个分组以独立白色圆角面板承载。
- 使用更清晰的标题层级，而不是靠粗边框做分隔。
- 面板内部继续承接原有入口，但整体样式需与主入口和辅助入口统一。

## 4. 信息架构与页面职责

首页的信息顺序固定为：

1. 顶部业务氛围与提醒
2. 投诉 / 报修双主入口
3. 巡检 / 设备等辅助入口
4. 业务分组面板

这套顺序背后的原则如下：

- 首页的第一任务是“让人尽快点进去”。
- 首页不负责承担完整看板职责。
- 首页不依赖摘要接口也能成立。
- 首页与工作台形成互补，而不是重复。

工作台继续承担以下职责：

- 更完整的功能分类索引
- 更强的个人身份感
- 更适合承载大量功能入口

## 5. 数据依赖策略

### 5.1. 当前阶段

为了避免被不确定接口阻塞，首页定稿方案默认采用低依赖策略：

- 顶部提醒文案可先使用静态常量。
- 业务入口直接沿用现有跳转能力。
- 不引入待办统计依赖。

### 5.2. 后续可选增强

如果未来补齐接口，可以在不破坏当前结构的前提下增强：

- 顶部提醒带切换为公告接口或运营配置接口。
- 针对不同角色输出不同提醒文案。
- 在主入口卡或业务面板内增加轻量数字角标。

### 5.3. 不建议在当前阶段引入的依赖

当前不建议将以下能力作为首页首屏成立的前置条件：

- 待处理总数
- 紧急件数量
- 进行中数量
- 任意需要聚合多个业务来源的摘要统计

## 6. 复用规则

这份设计稿不只服务首页，后续其他页面若需要向工作台风格靠齐，应优先复用以下规则：

- 使用蓝色渐变头部建立统一品牌氛围。
- 入口区优先使用白色圆角卡片，而不是旧式表格式宫格。
- 高优先级入口应使用更大的卡片尺寸和更明确的标签层级。
- 顶部如果没有稳定数据来源，优先使用轻公告带或业务说明带，不强行做摘要卡。
- 页面之间保持同源视觉语言，但允许根据职责调整首屏信息节奏。

换言之，后续的美术统一不应以“所有页面长得一样”为目标，而应以“同源视觉语言 + 不同职责节奏”为目标。

## 7. 实施边界

本设计稿只定义视觉与信息结构，不直接绑定具体实现细节。

实现时应遵守以下边界：

- 不改变现有业务路由语义。
- 不为实现这次设计而强行新增摘要接口。
- 不把首页改造成完整 dashboard。
- 不让顶部提醒带成为大段运营文案堆砌区。

## 8. 状态与兜底

为了保证方案可落地，需提前定义几个状态：

- 当顶部无动态文案时，显示默认业务提醒文案。
- 当某些入口仍为开发中状态时，保留禁用态表现，不破坏整体卡片布局。
- 当后续接入公告接口失败时，回退到静态文案，不影响首页结构。

## 9. 验收标准

如果后续实现符合以下标准，则视为达到本设计稿目标：

- 首页首屏能明显看出与工作台是同一视觉体系。
- 首页第一视觉重心是投诉、报修双主入口。
- 顶部区域不依赖摘要接口也能成立。
- 巡检、设备等入口仍可快速触达，但不会与主入口争夺焦点。
- 下方业务分组比旧版更有卡片感和层级感。
- 用户会把首页理解为“业务门户”，而不是“个人欢迎页”。

## 10. 后续用途

这份文档应作为后续页面美术统一时的基础参考稿。

适用场景如下：

- 首页后续实现或迭代
- 其他入口页向工作台风格对齐
- 后续页面需要讨论“是否适合做摘要头”时作为判断依据
- 未来补齐公告接口后，对顶部提醒带进行增强
