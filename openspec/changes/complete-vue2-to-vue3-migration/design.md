## Context

### 当前状态

本项目是基于 unibest 框架的智慧社区物业管理系统，正在进行从 Vue2 到 Vue3 的技术栈升级。根据 2026-03-03 的迁移进度报告：

- **总页面数**：140 个页面需要迁移
- **已完成迁移**：52 个页面（37%）
- **未完成迁移**：88 个页面（63%）
- **合规性问题**：已迁移的 52 个页面中有 20 个存在 Skills 合规性问题（通过率 62%）

### 技术背景

**源技术栈（Vue2 项目）**：
- Vue2 Options API
- ColorUI 样式框架
- uni-app 内置组件
- Java110Context + uni.request 接口调用
- pages.json 路由配置

**目标技术栈（Vue3 项目）**：
- Vue3 Composition API + TypeScript
- UnoCSS 原子化 CSS
- wot-design-uni 组件库
- Alova + TypeScript + Mock 接口
- 约定式路由

### 约束条件

1. **Skills 技能规范强制执行**：所有迁移必须严格遵循 `.claude/skills/` 目录下的 10 个技能文档
2. **子代理团队模式**：每个文件迁移都需要新建独立的子代理团队（技能指导、代码迁移、审核）
3. **质量优先**：不贪快，确保每个文件都符合 Skills 规范
4. **路由映射表**：必须严格按照 `docs/prompts/route-migration-map.yml` 的映射关系进行迁移

### 利益相关方

- **开发团队**：需要清晰的迁移任务清单和技术指导
- **项目负责人**：需要可追踪的迁移进度和质量保证
- **AI 子代理**：需要明确的任务分工和协作流程

## Goals / Non-Goals

### Goals

1. **完成 88 个未迁移页面**：按优先级（高→中→低）完成所有未迁移页面
2. **修复 20 个合规性问题**：确保已迁移页面 100% 符合 Skills 规范
3. **建立标准化迁移流程**：通过子代理团队模式确保迁移质量和效率
4. **实现进度可追踪**：每个任务都有明确的状态和完成标准

### Non-Goals

1. **不做功能增强**：仅做技术栈迁移，不添加新功能
2. **不做性能优化**：性能优化不在本次迁移范围内
3. **不做 UI 重设计**：保持原有 UI 设计，仅做组件替换
4. **不做登录鉴权**：项目不实现登录和鉴权功能（按 CLAUDE.md 第 3 条原则）

## Decisions

### 决策 1：采用子代理团队模式

**决策**：每个文件迁移都新建独立的 Agent Team，包含 3 个子代理成员。

**理由**：
- **质量保证**：技能指导子代理确保所有成员理解 Skills 规范
- **职责分离**：代码迁移子代理专注实现，审核子代理专注质量检查
- **可追溯性**：每个文件的迁移过程都有完整的记录

**替代方案**：
- ❌ 单一子代理完成所有工作：容易遗漏 Skills 规范，质量难以保证
- ❌ 手动迁移：效率低，难以保证一致性

### 决策 2：按模块优先级分阶段迁移

**决策**：按照高→中→低优先级顺序迁移，每次只处理一个模块。

**优先级排序**：
1. **高优先级**（43 个页面）：resource_modules（29 个）、fee_modules（14 个）
2. **中优先级**（24 个页面）：oa_modules（8 个）、property_modules（11 个）、parking_modules（5 个）
3. **低优先级**（21 个页面）：notice_modules（2 个）、basic_modules（5 个）、other_modules（14 个）

**理由**：
- **业务价值**：高优先级模块是核心业务功能（资源采购、费用管理）
- **风险控制**：分阶段迁移可以及时发现问题并调整策略
- **进度可见**：每完成一个模块都是明确的里程碑

**替代方案**：
- ❌ 并行迁移所有模块：容易失控，难以保证质量
- ❌ 按文件数量平均分配：忽略业务价值，不符合项目需求

### 决策 3：先修复合规性问题，再迁移新页面

**决策**：在开始迁移新页面之前，先修复已迁移页面的 20 个合规性问题。

**理由**：
- **建立标准**：修复过程可以帮助团队更好地理解 Skills 规范
- **避免重复工作**：如果先迁移新页面，可能会产生更多合规性问题
- **质量优先**：确保已完成的工作符合标准，再继续新工作

**替代方案**：
- ❌ 先迁移新页面，最后统一修复：可能导致大量返工
- ❌ 边迁移边修复：容易混乱，难以追踪进度

### 决策 4：使用细粒度任务清单

**决策**：为每个文件创建独立的任务项，包含具体的修改内容和 Skills 要求。

**任务清单结构**：
```
模块名称
  ├── 文件 1
  │   ├── 代码迁移（code-migration）
  │   ├── 组件迁移（component-migration）
  │   ├── 样式迁移（style-migration）
  │   ├── API 迁移（api-migration）
  │   ├── 路由迁移（route-migration）
  │   └── 合规性检查（use-wd-form、z-paging、api-error-handling 等）
  ├── 文件 2
  └── ...
```

**理由**：
- **清晰明确**：每个任务都有明确的输入、输出和验收标准
- **易于追踪**：可以实时查看每个文件的迁移状态
- **便于分配**：可以灵活地将任务分配给不同的子代理

**替代方案**：
- ❌ 粗粒度任务（按模块）：难以追踪具体进度
- ❌ 无任务清单：容易遗漏文件，难以保证完整性

### 决策 5：强制执行 Skills 技能触发检查

**决策**：每个文件迁移前，必须先阅读 `.claude/skills/check-trigger.md` 进行技能触发检查。

**理由**：
- **避免遗漏**：系统化的检查流程确保不会遗漏任何相关技能
- **多技能协同**：大多数任务需要多个技能协同（如表单页需要 use-wd-form + beautiful-component-design）
- **质量保证**：按照正确顺序执行技能，避免返工

**检查流程**：
1. 打开 `.claude/skills/check-trigger.md`
2. 逐项回答检查问题
3. 生成技能清单
4. 阅读所有相关技能的主文件和子文档
5. 创建任务清单
6. 开始执行

**替代方案**：
- ❌ 凭经验判断：容易遗漏技能，导致不合规
- ❌ 只使用部分技能：违反项目规范，需要返工

## Risks / Trade-offs

### 风险 1：子代理团队协作复杂度

**风险**：140 个文件需要创建 140 个子代理团队，协作复杂度高。

**缓解措施**：
- 使用标准化的团队模板和工作流程
- 每个团队只负责一个文件，完成后立即删除
- 技能指导子代理可以在多个团队之间复用

### 风险 2：Skills 规范理解偏差

**风险**：不同子代理对 Skills 规范的理解可能存在偏差，导致不一致。

**缓解措施**：
- 技能指导子代理负责统一解释 Skills 规范
- 审核子代理严格按照 Skills 规范检查
- 发现问题立即修正，避免错误传播

### 风险 3：迁移进度延期

**风险**：88 个页面 + 20 个修复任务，工作量大，可能延期。

**缓解措施**：
- 按优先级分阶段迁移，确保核心功能优先完成
- 每个阶段设置明确的里程碑和验收标准
- 定期检查进度，及时调整策略

### 风险 4：合规性检查遗漏

**风险**：可能遗漏某些 Skills 规范要求，导致后期返工。

**缓解措施**：
- 强制执行 `.claude/skills/check-trigger.md` 检查流程
- 审核子代理使用多重搜索策略（正则、宽泛关键词、相关标签）
- 完成后进行全面的合规性检查

### Trade-off 1：质量 vs 速度

**选择**：优先保证质量，不贪快。

**理由**：
- 返工成本远高于一次做对的成本
- 合规性问题会影响后续开发效率
- 项目处于开发阶段，有足够时间保证质量

### Trade-off 2：自动化 vs 人工审核

**选择**：采用子代理自动化迁移 + 人工审核的混合模式。

**理由**：
- 自动化提高效率，减少重复劳动
- 人工审核确保质量，发现自动化难以发现的问题
- 混合模式平衡效率和质量

### Rollback 策略

由于本项目使用 Git 版本控制，回滚策略如下：

1. **单文件回滚**：使用 `git checkout <commit> -- <file>` 回滚单个文件
2. **模块回滚**：使用 `git revert <commit>` 回滚整个模块的提交
3. **分支回滚**：如果整个阶段出现问题，可以切换回上一个稳定分支

**注意**：每个阶段完成后都应该创建 Git 标签，便于回滚。

## Open Questions

### Q1：是否需要并行迁移多个模块？

**当前决策**：按顺序迁移，每次只处理一个模块。

**待讨论**：如果项目进度紧张，是否可以并行迁移 2-3 个模块？

### Q2：合规性检查的自动化程度？

**当前决策**：使用子代理进行自动化检查 + 人工审核。

**待讨论**：是否需要开发自动化检查工具（如 ESLint 插件）？

### Q3：迁移过程中发现 Skills 规范不完善怎么办？

**当前决策**：暂停迁移，先完善 Skills 规范，再继续迁移。

**待讨论**：是否需要建立 Skills 规范的更新流程？

### Q4：如何处理迁移过程中发现的 Bug？

**当前决策**：记录 Bug，但不在本次迁移中修复（除非影响迁移）。

**待讨论**：是否需要建立 Bug 跟踪机制？
