# 2026-04-04 微信小程序个人主体无法使用 web-view 套壳的经验记录

## 1. 问题现象

- 计划采用“微信小程序轻量壳 + H5 主业务”的方式推进，并已补出 `web-view` 容器页、H5 域名配置和首页入口。
- 实际推进到微信后台配置阶段时，发现当前小程序账号主体为个人主体，无法把方案真正落地为可上线、可验收的 `web-view` 套壳链路。

## 2. 根因

- 真正的阻塞点不是代码未打通，也不是域名未备案，而是产品前提不成立：
  1. 当前账号主体是个人主体小程序。
  2. 该方案依赖 `web-view` 业务域名能力承载 H5 主业务。
  3. 个人主体不具备这条可交付能力前提，因此整个“小程序壳 + H5 主业务”方案在当前账号下不成立。

## 3. 关键误导点

- 误导点 1：先从技术实现入手，容易误以为“只要补齐域名、环境变量、容器页，链路就能跑通”。
- 误导点 2：看到微信后台已经能配置 `request 合法域名`，容易误判为 `web-view` 套壳也只是域名配置问题。
- 误导点 3：在没有先核对账号主体能力边界前，就开始实现 `web-view` 壳层，会把大量时间投入到一个账号前提就不支持的方向上。

## 4. 有效修复

- 本次有效修复不是继续补代码，而是承认方案前提不成立并立即止损：
  1. 回退本轮 `web-view` 套壳相关实现，包括 H5 壳地址解析、首页入口、容器页升级、配套测试和投诉链路缓存补丁。
  2. 保留计划文档与经验记录，避免后续 agent 再在个人主体账号上重复推进同一路线。
  3. 明确后续可选方向只有三类：
     - 更换为非个人主体小程序，继续推进 `web-view` 套壳方案。
     - 放弃小程序壳，转为“微信内直接访问 H5”。
     - 回退为原生 uni-app 小程序页面方案。

## 5. 验证方式

- 验证回退是否完成：
  1. `git status --short` 中不应再出现本轮 `web-view` 套壳相关文件变更。
  2. 首页不再保留 H5 壳入口，代码库不再保留 `src/utils/mp-weixin-h5-shell.ts`、`src/pages/webview/index.vue` 的本轮升级逻辑及配套测试。
- 验证经验结论是否成立：
  1. 先核对当前小程序账号主体类型。
  2. 若仍为个人主体，则禁止再把 `web-view` 套壳当成当前账号下的交付路径。

## 6. 后续约束

- 以后只要涉及“小程序内嵌 H5 / web-view 套壳”，必须先检查下面两个前提，再决定是否编码：
  1. 当前小程序主体是否为非个人主体。
  2. 当前主体是否具备所需的 `web-view` / 业务域名能力。
- 如果主体是个人主体：
  1. 不要继续推进 `web-view` 套壳实现。
  2. 先和用户确认是换主体，还是切换到“微信内 H5”或“原生小程序页”方案。
- 以后遇到“技术方案天然依赖平台能力”的任务，优先先核对账号/主体/权限边界，再做实现。
